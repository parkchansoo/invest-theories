## 1. 한국 공시 체계 전체 구조

한국 상장기업의 공시 의무는 **3개의 독립된 규제기관**에 의해 부과되며, 모든 공시는 **DART(전자공시시스템)** 에 통합 게재된다.

```
                        ┌─────────────────────────┐
                        │   DART (dart.fss.or.kr)  │
                        │   전자공시시스템 통합 게재  │
                        └────────┬────────────────┘
                                 │
              ┌──────────────────┼──────────────────┐
              │                  │                   │
   ┌──────────▼──────┐  ┌───────▼────────┐  ┌──────▼──────────┐
   │  금감원 (FSS)    │  │  거래소 (KRX)   │  │  공정위 (KFTC)   │
   │  자본시장법       │  │  거래소 공시규정  │  │  공정거래법       │
   │                  │  │                │  │                  │
   │  A: 정기공시      │  │  I: 거래소공시   │  │  J: 공정위공시    │
   │  B: 주요사항보고   │  │   I001 수시공시  │  │   J001 내부거래   │
   │  C: 발행공시      │  │   I002 공정공시  │  │   J004 집단현황   │
   │  D: 지분공시      │  │   I003 시장조치  │  │   J005 비상장중요  │
   │  E: 기타공시      │  │   I004 지분공시  │  │   J006 기타      │
   │  F: 외부감사      │  │   I005 증권투자  │  │   J008 공익법인   │
   │  G: 펀드공시      │  │   I006 채권공시  │  │   J009 하도급     │
   │  H: 자산유동화    │  │                │  │                  │
   │                  │  │                │  │                  │
   │  구조화API: 있음   │  │  구조화API: 없음 │  │  구조화API: 없음   │
   │  (DS001~DS006)   │  │  (목록만 조회)   │  │  (목록만 조회)     │
   └──────────────────┘  └────────────────┘  └──────────────────┘
```

### 1.1 세 플랫폼 비교

| | 금감원 (FSS) | 거래소 (KRX) | 공정위 (KFTC) |
|---|---|---|---|
| **근거법** | 자본시장법 | 유가증권시장/코스닥시장 공시규정 | 공정거래법 (독점규제 및 공정거래에 관한 법률) |
| **대상** | 모든 상장법인 + 사업보고서 제출대상 | 상장법인 | 자산총액 5조원 이상 기업집단 소속 회사 |
| **대표 공시** | 증자, CB/BW, 합병, 분할, 감자, 재무제표 | 수주, 배당결정, 잠정실적, 대표이사변경 | 대규모내부거래, 기업집단현황, 비상장회사중요사항 |
| **DART 게재** | O (원본) | O (연동 게재) | O (연동 게재) |
| **OpenDART 코드** | `pblntf_ty` = A, B, C, D, E, F, G, H | `pblntf_ty` = I | `pblntf_ty` = J |
| **구조화 API** | DS001~DS006 (79개) | 없음 | 없음 |
| **별도 포털** | opendart.fss.or.kr | kind.krx.co.kr | egroup.go.kr (기업집단포털) |

### 1.2 데이터 접근 경로

```
데이터 수집 관점에서:

[구조화 JSON]  OpenDART DS001~DS006 API
               → 금감원 공시의 필드별 데이터 직접 획득
               → 증자, CB, 합병, 재무제표, 지분공시 등

[목록 + 원문]  OpenDART 공시검색 API (list)
               → 금감원 + KRX + 공정위 공시 모두 목록 조회 가능
               → pblntf_ty 파라미터로 플랫폼 필터
               → 내용물은 공시서류원본파일(document) API → HTML 파싱

[웹 열람 전용]  KIND (kind.krx.co.kr) — KRX 공시 웹 열람
               egroup.go.kr — 공정위 기업집단 공시 웹 열람
```

## 2. OpenDART API 구조

**Base URL**: `https://opendart.fss.or.kr/api/`  
**인증**: API Key 40자리, 일 20,000건 제한  
**포맷**: JSON / XML  
**데이터 범위**: 2015년 이후 (주요사항보고서 기준)

### 2.1 구조화 API 그룹 (DS001~DS006)

| 그룹 | 코드 | API 수 | 내용 |
|---|---|---|---|
| 공시정보 | DS001 | 4 | 공시검색, 기업개황, 원본파일, 고유번호 |
| 정기보고서 주요정보 | DS002 | 28 | 사업/분기/반기보고서 내 추출 데이터 |
| 정기보고서 재무정보 | DS003 | 7 | XBRL 재무제표, 주요계정, 재무지표 |
| 지분공시 종합정보 | DS004 | 2 | 대량보유, 임원/주요주주 변동 |
| 주요사항보고서 주요정보 | DS005 | 36 | 이벤트성 주요사항 보고서 |
| 증권신고서 주요정보 | DS006 | 6 | 지분/채무증권, 합병/분할 신고서 |
| **합계** | | **83** | |

### 2.2 공시검색 API

모든 공시의 진입점. **금감원·KRX·공정위 공시 전체를 통합 검색**할 수 있다.

```
GET https://opendart.fss.or.kr/api/list.json
  ?crtfc_key={KEY}
  &bgn_de=20240101        // 시작일 (YYYYMMDD)
  &end_de=20241231        // 종료일
  &pblntf_ty=I            // 공시유형 (A~J)
  &pblntf_detail_ty=I001  // 상세유형
  &corp_cls=Y             // 법인구분 (Y=유가, K=코스닥, N=코넥스, E=기타)
  &last_reprt_at=Y        // 최종보고서만 (정정 있으면 최종정정만)
  &page_count=100         // 페이지당 건수 (최대 100)
```

#### 응답 필드

| 필드 | 설명 | 비고 |
|---|---|---|
| `corp_cls` | 법인구분 | Y(유가), K(코스닥), N(코넥스), E(기타) |
| `corp_name` | 종목명(법인명) | |
| `corp_code` | 고유번호 | 8자리. 모든 상세 API의 키 |
| `stock_code` | 종목코드 | 6자리. 상장사만 |
| `report_nm` | 보고서명 | `[기재정정]`, `[첨부정정]` 등 접두사 포함 |
| `rcept_no` | 접수번호 | 14자리. 공시 원문 링크에 사용 |
| `flr_nm` | 제출인명 | |
| `rcept_dt` | 접수일자 | YYYYMMDD |
| `rm` | 비고 | 유/코/채/넥/공/연/정/철 조합 |

#### `rm` (비고) 필드 의미

| 문자 | 의미 |
|---|---|
| 유 | 한국거래소 유가증권시장 소관 |
| 코 | 한국거래소 코스닥시장 소관 |
| 넥 | 한국거래소 코넥스시장 소관 |
| 채 | 한국거래소 채권상장법인 소관 |
| 공 | 공정거래위원회 소관 |
| 연 | 연결부분 포함 |
| 정 | 정정신고 있음 (관련 보고서 참조) |
| 철 | 철회(간주)됨 |

| **P2** | 경영권·구조 변동 | 구조화 API | 대량보유, 내부자거래, 영업양수도, 소송, 위기이벤트 (~10) | 경영권 이슈 조기 포착 |
| **P3** | 재무 펀더멘탈 | 구조화 API | 재무제표, 주요계정, 재무지표, 감사의견 (~7) | 밸류에이션, 실적 분석 |
| **P4** | 주주환원·배당 | 구조화 API | 배당, 자사주 현황 (~4) | 인컴 전략, 주주환원 추적 |
| **P5** | 거버넌스 | 구조화 API | 최대주주, 임원, 보수, 소액주주 (~8) | ESG, 스튜어드십 |
| **P6** | KRX 수시·공정공시 | **HTML 파싱** | 수주, 배당결정, 잠정실적, 대표이사변경, 시설투자 (~8종) | 일반 투자자 핵심 공시 |
| **P7** | 공정위 공시 | 목록 + 원문 | 대규모내부거래, 기업집단현황, 비상장중요사항 (~6종) | 대기업 지배구조 분석 |
| **P8** | 발행·기타·감사 | 구조화 API + 목록 | 증권신고서, 자기주식신고, 주총소집, 감사보고서 | 보조적, 배치 가능 |

### 4.2 수집 주기 가이드

| 주기 | 대상 | 이유 |
|---|---|---|
| **실시간** (5~15분 폴링) | P1 (증자, CB, 합병 등), P6 중 수주·잠정실적 | 공시 즉시 주가 반응 |
| **준실시간** (30분~1시간) | P2 (대량보유, 소송), P6 나머지, P7 | 중기 영향 |

| **분기 배치** | DS002 정기보고서 주요정보 (28개 API) | 분기 보고서 제출 후 1회 |

## 5. 정정공시 처리 전략

정정공시는 데이터 파이프라인에서 **가장 까다로운 이슈**이다. 잘못 처리하면 원본과 정정본이 중복 적재되거나, 정정된 데이터가 누락된다.

### 5.1 정정공시의 구조

```
[원본]     rcept_no=20240315000001   report_nm="유상증자결정"
  ↓ 정정
[정정1]    rcept_no=20240320000015   report_nm="[기재정정]유상증자결정"
  ↓ 재정정
[정정2]    rcept_no=20240325000008   report_nm="[기재정정]유상증자결정"
```

- **원본과 정정본은 서로 다른 `rcept_no`를 가진다** (별도의 보고서)
- 원본의 `rm` 필드에 `정`이 추가된다 (정정신고 있음 표시)
- 정정본의 `report_nm`에 `[기재정정]`, `[첨부정정]`, `[첨부추가]` 등 접두사가 붙는다

### 5.2 `report_nm` 접두사 유형

| 접두사 | 의미 | 처리 방법 |
|---|---|---|
| `[기재정정]` | 본문 내용이 변경됨 | **데이터 갱신 필요. 원본 대체** |
| `[첨부정정]` | 첨부파일이 변경됨 | 첨부 의존 시 갱신. 아니면 무시 가능 |
| `[첨부추가]` | 첨부파일 추가 | 보통 무시 가능 |
| `[변경등록]` | 유동화계획 변경 | ABS 관련 |
| `[연장결정]` | 신탁계약 연장 | 신탁 관련 |
| `[발행조건확정]` | 유가증권 발행조건 확정 | 증권신고서 관련 |
| `[정정명령부과]` | 금감원 정정명령 | 규제 리스크 시그널 |
| `[정정제출요구]` | 금감원 정정제출요구 | 규제 리스크 시그널 |

### 5.3 철회 처리

| `rm` 값 | 의미 | 처리 |
|---|---|---|
| `철` | 보고서 철회(간주)됨 | **해당 공시 데이터를 무효 처리해야 함** |

### 5.4 `last_reprt_at` 파라미터

```
last_reprt_at=Y  →  정정이 있는 경우 최종 정정본만 반환
last_reprt_at=N  →  원본 + 모든 정정본 모두 반환 (기본값)
```

> **실무 주의**: `last_reprt_at=Y`는 **정기보고서(A유형: 사업/반기/분기보고서)에만 안정적으로 동작**한다. 수시공시(B: 주요사항보고, I: 거래소공시 등)에서는 원본이 포함되거나 정정본이 누락되는 케이스가 실무에서 보고되고 있다. 공식 문서에 이 제한이 명시되어 있지 않아 주의가 필요하다.

### 5.5 적재 전략 권장 패턴

#### 패턴 A: 최신 상태만 필요한 경우 (뉴스 피드, 푸시 알림)

```
1. 정기공시(A): last_reprt_at=Y 사용 가능
2. 수시공시(B,I,J): last_reprt_at=N으로 전부 수집 후 직접 최신본 필터링
   - report_nm 파싱 → correction_type 추출
   - (corp_code, base_report_nm, year) 그룹 내 가장 늦은 rcept_dt가 최종
3. rcept_no 기준 upsert
4. 장점: 단순
5. 단점: 정정 히스토리 소실, 수시공시는 수동 필터링 필요
```

#### 패턴 B: 변화 추적이 필요한 경우 (히스토리컬 분석)

```
1. last_reprt_at=N 로 조회 → 원본 + 정정 전부 수집
2. 같은 공시에 대한 원본↔정정 연결 로직 필요:
   - corp_code + report_nm(접두사 제거) + 날짜 근접성으로 그룹핑
   - 또는 구조화 API(DS005 등)에서 같은 corp_code + 기간으로 조회 시
     정정 전/후 데이터를 모두 확보 → diff 가능
3. 각 레코드에 version 필드 부여 (v1=원본, v2=정정1, ...)
4. is_latest 플래그로 최종본 표시
5. 장점: 정정 이력 전체 보존, "무엇이 바뀌었는지" 분석 가능
6. 단점: 복잡도 증가, 연결 로직 유지보수 필요
```

#### 패턴 C: 혼합 (권장)

```
- 구조화 API(DS002~DS006): last_reprt_at=Y 로 최종본만 수집 (기본)
  - 필요 시 특정 보고서만 N으로 재조회하여 diff
- 공시검색 목록(list): last_reprt_at=N 으로 전부 수집
  - report_nm 파싱으로 원본/정정 구분
  - 푸시 알림 시: [기재정정] 접두사 있으면 "정정공시" 라벨 추가
```

## 6. 적재 시 실무 유의사항

### 6.1 정정공시 모니터링 윈도우

공시 수집 후 **수일~수주 뒤에 정정이 올 수 있다**. 특히 정기보고서의 XBRL 주석은 외주 작성 기업의 **재공시율이 28~43%** 에 달한다 (DartLab 분석, 2026).

```
권장: 최초 수집 후 최소 60일간 해당 기업의 정정공시를 모니터링하는 별도 잡 운영.
정정 발견 시: 이전 버전 superseded 처리 + [정정공시 발생] 알림.
```

#### 권장 DB 스키마

```sql
CREATE TABLE disclosures (
    rcept_no        VARCHAR(14) PRIMARY KEY,
    corp_code       VARCHAR(8)  NOT NULL,
    report_nm       VARCHAR(200),
    base_report_nm  VARCHAR(200),  -- 접두어 제거한 기본 보고서명
    correction_type VARCHAR(10),   -- NULL, '기재정정', '첨부정정', '철회' 등
    rcept_dt        DATE,
    is_superseded   BOOLEAN DEFAULT FALSE,  -- 더 최신 정정본이 있으면 TRUE
    superseded_by   VARCHAR(14),            -- 대체한 rcept_no
    is_active       BOOLEAN DEFAULT TRUE,   -- 철회 시 FALSE
    INDEX idx_corp_base (corp_code, base_report_nm, rcept_dt)
);

-- 최신 유효 공시만 조회하는 뷰
CREATE VIEW latest_disclosures AS
SELECT * FROM disclosures
WHERE is_superseded = FALSE AND is_active = TRUE;
```

| 항목 | 제한 |
|---|---|
| 일일 호출 한도 | 20,000건 |
| 페이지당 최대 건수 | 100건 |

**전략**: 공시검색 API로 "새 공시 있는 종목"만 선별 → 해당 종목만 상세 API 호출. 전 종목 순회하면 일일 한도 금방 소진됨.

### 6.3 고유번호(corp_code) vs 종목코드(stock_code)

```
- OpenDART의 모든 API는 corp_code (8자리)를 키로 사용
- 종목코드(6자리)는 상장사만 보유. 비상장법인은 stock_code 없음
- 고유번호 파일(corpCode.xml)을 먼저 다운로드하여 매핑 테이블 구축 필수
- 같은 회사명이 ~7% 중복 존재 → corp_code로만 식별해야 함
```

### 6.4 데이터 시작 시점

| API 그룹 | 데이터 시작 |
|---|---|
| 공시검색 (list) | DART 서비스 개시 이후 전체 (~1999년) |
| DS002~DS006 구조화 API | **2015년 이후** |
| 재무제표 (DS003) | XBRL 적용 이후 (~2011년, 법인에 따라 상이) |

### 6.5 corp_code 없이 검색할 때

```
corp_code가 비어있으면 검색기간은 3개월로 제한됨.
전 종목 공시를 수집하려면:
  - 기간을 3개월 이하로 잘라서 반복 호출
  - 또는 corp_code별로 호출 (더 정확하지만 호출 수 많음)
```

### 6.6 KRX 공시(I001, I002) 원문 파싱 시 주의

```
1. 공시 양식이 표준화되어 있지 않음
   - 같은 "단일판매·공급계약체결"이라도 회사마다 HTML 테이블 구조 상이
   - 코스피 vs 코스닥 양식 차이 있음

2. 공시서류원본파일 API 응답은 ZIP
   - ZIP 내부에 XML(목차) + 다수 HTML 파일
   - 본문이 여러 HTML에 걸쳐 분산될 수 있음

3. 금액 필드 파싱 이슈
   - "1,234,567" 또는 "1234567" 또는 "12억3456만" 등 혼재
   - null, "-", "해당사항 없음", 공백 등 다양한 빈 값 표현
   - 단위가 "원", "백만원", "억원" 등 혼재

4. 자회사 공시
   - 자회사의 수주/주요경영사항은 모회사명 + 자회사명 병기
   - 파싱 시 어느 법인의 공시인지 구분 필요
```

### 6.7 공정위 공시(J) 파싱 시 주의

```
1. 대상이 한정적 (자산 5조원 이상 기업집단 소속사만, 약 88~92개 집단)
   - 전체 상장사 대비 소수이므로 볼륨은 크지 않음

2. 기업집단현황공시(J004)는 연 1회(매년 5월) 대량 데이터
   - 기업집단 단위로 제출됨 → 개별 계열사 corp_code가 아닌 기업집단 대표사 코드로 검색 필요
   - 계열사 수십~수백 개의 출자관계, 채무보증 포함
   - 그래프 구조 데이터 → 관계형 테이블 설계 필요

3. 비상장회사중요사항(J005)
   - 비상장사는 stock_code가 없음
   - corp_code로만 식별. 매핑 테이블에서 누락될 수 있음

4. 하도급대금결제조건공시(J009)는 2020년 이후 신설
   - 과거 데이터 없음. 반기별(1월, 7월) 수집

5. 공정위 공시의 고유 가치
   - 비상장 계열사 재무 요약 (J004) → 개별 사업보고서에서는 얻을 수 없는 정보
   - 이사회 의결 전 내부거래 사전 공시 (J001) → 사업보고서는 연간 사후 집계뿐
   - 공익법인 지분 현황 (J004, J008) → 사업보고서 미포함

6. 공공데이터포털(data.go.kr) 보조 활용
   - "공정거래위원회 기업집단" 검색 → 정형 데이터 API 제공
   - 대규모기업집단 소속회사 주주현황, 임원현황, 지정현황, 계열사현황 등
   - 연간 스냅샷이므로 실시간성은 낮으나, 마스터 데이터로 활용 가능
```

### 6.8 동일 공시 히스토리컬 변화 추적

제품 목표인 "동일 공시 내 히스토리컬 변화 추적"을 위한 설계 포인트:

```
1. 구조화 API (DS005 등)
   - 같은 corp_code + 같은 API + 다른 기간으로 조회하면
     해당 기업의 해당 유형 공시 이력 전체를 확보 가능
   - 예: 삼성전자의 전환사채 발행 이력 → cvbdIsDecsn API에
     corp_code=00126380, bgn_de=20150101, end_de=20261231

2. 비구조화 공시 (KRX, 공정위)
   - 공시검색 API에서 corp_code + 키워드(report_nm LIKE)로
     같은 유형의 공시 이력 추출
   - 원문 파싱 결과를 시계열로 적재하면
     "이 회사의 수주금액 추이", "배당금 변화" 등 분석 가능

3. 키 설계
   - (corp_code, disclosure_type, rcept_dt, rcept_no) 복합 키 권장
   - disclosure_type은 자체 정의한 유형 코드 (pblntf_detail_ty 또는 커스텀)
```

### 6.9 투자정보 추출 관점

```
1. 구조화 API에서 직접 추출 가능한 정보
   - 증자 규모 vs 기존 발행주식 → 희석율
   - CB 전환가 vs 현재가 → 오버행 규모
   - 합병비율 + 합병상대 재무 → 합병 적정성
   - 자사주 취득 규모 vs 시가총액 → 환원율

2. HTML 파싱 후 추출 가능한 정보
   - 수주금액 vs 매출액 → 수주잔고 비율
   - 잠정실적 vs 컨센서스 → 서프라이즈율
   - 배당금 vs 전년 → 배당 성장률

3. 크로스 소스 분석
   - 자사주 취득 결정(DS005) + 자기주식 현황(DS002) → 실제 이행률
   - 유상증자(DS005) + 공모자금 사용내역(DS002) → 자금 용도 이행
   - 대량보유(DS004) + 주가 데이터(외부) → 5% 보고 전후 주가 반응
```

### 6.10 뉴스 데이터로 활용 시 고려사항

```
1. report_nm을 자연어 제목으로 가공
   - "유상증자결정" → "삼성전자, 유상증자 1,000억원 결정"
   - 구조화 필드에서 금액·수량 추출하여 제목에 반영

2. 공시 중요도 라벨링
   - P1~P2: "긴급", "주가영향"
   - P3~P5: "재무", "거버넌스"
   - P6: "수시", "실적"
   - P7: "대기업집단"

3. 중복 제거 → 아래 "7. 이벤트 그룹핑" 참조
```

## 7. 공시 간 중복과 이벤트 그룹핑

### 7.1 동일서식 공시의 DART↔KIND 관계

주요사항보고서(B유형) 중 거래소 수시공시와 동일서식을 사용하는 항목은 기업이 **DART에 1회 제출**하면 KIND로 자동 전송된다.  
그러나 DART 내부에서는 **B유형(금감원 직접)과 I유형(KIND 경유)으로 각각 별도 `rcept_no`가 생성**된다.

```
[동일서식 항목] — 기업이 DART에 1건 제출하지만, 내부적으로 2건 등재

  rcept_no = 20240315 001234   ← 금감원 직접 (일련번호 0xxxxx)
  pblntf_ty = B
  report_nm = "주요사항보고서(유상증자결정)"

  rcept_no = 20240315 801234   ← KIND 경유 (일련번호 8xxxxx)
  pblntf_ty = I
  report_nm = "유상증자결정"

  → 같은 이벤트, 같은 날, 서로 다른 rcept_no 2개

[KRX 고유 항목] — 금감원 보고 의무 없는 거래소 공시

  rcept_no = 20240315 8xxxxx   ← KIND 경유만
  pblntf_ty = I 만
  report_nm = "단일판매ㆍ공급계약체결", "현금ㆍ현물배당결정" 등
```

**핵심**: `rcept_no` 일련번호 패턴으로 출처를 구분할 수 있다.

| 일련번호 범위 | 출처 | pblntf_ty |
|---|---|---|
| `0xxxxx` (6자리, 0으로 시작) | 금감원 직접 제출 | A, B, C, D, E, F |
| `8xxxxx` (6자리, 8로 시작) | KIND 경유 (거래소공시) | I |

**따라서 동일서식 항목은 B+I 중복이 발생한다. 이벤트 그룹핑으로 중복 제거 필요.**

### 7.2 같은 이벤트가 여러 공시로 나오는 패턴

동일 문서 중복은 아니지만, **같은 기업 이벤트가 시차를 두고 여러 유형의 공시**를 발생시킨다.

#### 유상증자 이벤트 — 최대 5건의 공시

```
시간순 →

[T+0일] B유형: 유상증자결정 (주요사항보고서)
        └ report_nm = "유상증자결정"
        └ DS005 구조화 API 있음 (piicDecsn)

[T+수일] C유형: 증권신고서(지분증권) (발행공시)
         └ report_nm = "증권신고서(지분증권)"
         └ DS006 구조화 API 있음

[T+수일] C유형: 투자설명서
         └ report_nm = "투자설명서"

[T+수주] E유형: 주주총회소집보고서
         └ 증자 관련 주총 안건

[T+수개월] A유형: 사업보고서 내 "공모자금의 사용내역"
           └ DS002 구조화 API (공모자금사용내역)
```

#### 자기주식 취득 이벤트 — 3건

```
[T+0일] B유형: 자기주식취득결정 (주요사항보고서)
        └ report_nm = "자기주식취득결정"
        └ DS005 (tsstkAqDecsn)

[T+수일~수개월] E유형: 자기주식취득/처분 (기타공시)
               └ report_nm = "자기주식취득상황보고서"
               └ 실제 매수 진행 상황 보고

[T+분기말] A유형: 분기보고서 내 "자기주식 취득 및 처분 현황"
           └ DS002 (tsstkAqDsp)
```

#### 합병 이벤트 — 5건 이상

```
[T+0일]   B유형: 회사합병결정 (주요사항보고서)
[T+수일]  C유형: 증권신고서(합병등)
[T+수주]  E유형: 주주총회소집보고서
[T+수개월] E유형: 합병등종료보고서
[T+수개월] A유형: 합병 후 최초 사업보고서
```

#### 배당 이벤트 — 2건 (B↔I 분리)

```
[T+0일]   I유형: 현금·현물배당결정 (거래소 수시공시)
          └ report_nm = "현금·현물배당결정"
          └ KRX 고유. B유형에 해당 없음.

[T+수개월] A유형: 사업보고서 내 "배당에 관한 사항"
           └ DS002 (dvRs)
```

#### 7.3.4 report_nm 정규화 규칙

`report_nm`에서 접두사를 제거하고 핵심 보고서명만 추출하여 event_type 매핑에 사용한다.

**B↔I 보고서명 관계**: B유형의 괄호 안 텍스트 = I유형의 보고서명 전체
```
B유형: "주요사항보고서(유상증자결정)"  → 괄호 안 추출 → "유상증자결정"
I유형: "유상증자결정"                → 그대로 사용

B유형: "주요사항보고서(전환사채권발행결정)" → "전환사채권발행결정"
I유형: "전환사채권발행결정"
```

```python
import re

# 1. 정정/철회 접두사 제거
CORRECTION_PREFIX = re.compile(r'^\[(기재정정|첨부정정|첨부추가|정정|철회|철회간주)\]')

# 2. 자회사 접미사 추출
SUBSIDIARY_SUFFIX = re.compile(r'\(자회사의 주요경영사항\)$')

# 3. report_nm → event_type 매핑 사전
REPORT_NM_TO_EVENT = {
    "유상증자결정": "CAPITAL_INCREASE",
    "무상증자결정": "CAPITAL_INCREASE_FREE",
    "전환사채권발행결정": "CB_ISSUANCE",
    "신주인수권부사채권발행결정": "BW_ISSUANCE",
    "교환사채권발행결정": "EB_ISSUANCE",
    "자기주식취득결정": "TREASURY_STOCK_BUY",
    "자기주식처분결정": "TREASURY_STOCK_SELL",
    "회사합병결정": "MERGER",
    "회사분할결정": "SPLIT",
    "회사분할합병결정": "SPLIT",
    "감자결정": "CAPITAL_DECREASE",
    "증권신고서(지분증권)": "CAPITAL_INCREASE",
    "증권신고서(채무증권)": "CB_ISSUANCE",  # 또는 BW
    "증권신고서(합병등)": "MERGER",
    "합병등종료보고서": "MERGER",
    "자기주식취득상황보고서": "TREASURY_STOCK_BUY",
    "단일판매ㆍ공급계약체결": "CONTRACT_NEW",
    "단일판매ㆍ공급계약해지": "CONTRACT_CANCEL",
    "현금ㆍ현물배당결정": "DIVIDEND_CASH",
    "주식배당결정": "DIVIDEND_STOCK",
    "영업(잠정)실적(공정공시)": "EARNINGS_PRELIM",
    "주식등의대량보유상황보고서": "LARGE_HOLDING",
    # ... 확장 가능
}

def normalize_report_nm(report_nm: str) -> dict:
    # 정정 접두사 제거
    clean = CORRECTION_PREFIX.sub('', report_nm).strip()
    
    # 자회사 접미사 분리
    is_subsidiary = bool(SUBSIDIARY_SUFFIX.search(clean))
    clean = SUBSIDIARY_SUFFIX.sub('', clean).strip()
    
    # event_type 매핑
    event_type = REPORT_NM_TO_EVENT.get(clean, "UNKNOWN")
    
    return {
        "original": report_nm,
        "base_name": clean,
        "event_type": event_type,
        "is_subsidiary": is_subsidiary,
    }
```

#### 7.3.5 rcept_no 구조 활용

```
rcept_no = "20240315000123"
             ^^^^^^^^------
             YYYYMMDD  6자리 일련번호

일련번호 범위:
  0xxxxx = 금감원 직접 제출 (A, B, C, D, E, F 유형)
  8xxxxx = KIND 경유 거래소공시 (I 유형)
```

- 앞 8자리가 접수일자이므로, `rcept_dt` 필드와 동일한 날짜 정보
- **일련번호 첫 자리로 금감원/거래소 공시 구분 가능**
- 동일서식 항목(유상증자 등)은 같은 날 `0xxxxx`(B유형)과 `8xxxxx`(I유형) 쌍으로 등재됨

```python
def is_kind_filing(rcept_no: str) -> bool:
    """KIND 경유 거래소공시 여부"""
    return int(rcept_no[8:]) >= 800000

def find_b_i_pairs(filings_same_day_same_corp: list[dict]) -> list[tuple]:
    """같은 날 같은 기업의 B↔I 쌍 매칭"""
    fss = [f for f in filings_same_day_same_corp
           if f["pblntf_ty"] == "B" and not is_kind_filing(f["rcept_no"])]
    kind = [f for f in filings_same_day_same_corp
            if f["pblntf_ty"] == "I" and is_kind_filing(f["rcept_no"])]
    
    pairs = []
    for b in fss:
        # B type "주요사항보고서(유상증자결정)" → 괄호 안 "유상증자결정"
        match = re.search(r'\((.+?)\)$', b["report_nm"])
        if not match:
            continue
        keyword = match.group(1)
        
        for i in kind:
            # I type "유상증자결정" = 괄호 안 텍스트와 일치
            base_i = CORRECTION_PREFIX.sub('', i["report_nm"]).strip()
            if base_i == keyword:
                pairs.append((b, i))
    
    return pairs
```

#### 7.3.6 대표 공시 선정 로직

같은 이벤트 그룹 내에서 뉴스 피드·푸시에 노출할 **대표 공시 1건**을 선정하는 로직:

```python
# 우선순위: B(결정) > C(신고서) > E(완료) > I(거래소) > A(정기)
TYPE_PRIORITY = {"B": 1, "C": 2, "E": 3, "I": 4, "D": 5, "J": 6, "A": 7}

def select_representative(group: list[dict]) -> dict:
    """이벤트 그룹 내에서 대표 공시 선정"""
    # 1. 정정/철회 제외 (최종본만)
    active = [f for f in group if not f.get("is_superseded") and f.get("is_active")]
    
    # 2. pblntf_ty 우선순위로 정렬 (B가 가장 높음)
    active.sort(key=lambda f: (
        TYPE_PRIORITY.get(f["pblntf_ty"], 99),
        f["rcept_dt"]  # 같은 유형 내에서는 가장 빠른 것
    ))
    
    return active[0] if active else group[0]
```

### 7.4 이벤트 그룹핑 DB 스키마

```sql
-- 이벤트 그룹 테이블
CREATE TABLE disclosure_events (
    event_id        BIGINT AUTO_INCREMENT PRIMARY KEY,
    corp_code       VARCHAR(8)   NOT NULL,
    event_type      VARCHAR(30)  NOT NULL,     -- CAPITAL_INCREASE, MERGER 등
    event_date      DATE         NOT NULL,     -- 최초 공시일
    representative_rcept_no VARCHAR(14),        -- 대표 공시
    is_subsidiary   BOOLEAN DEFAULT FALSE,
    INDEX idx_corp_event (corp_code, event_type, event_date)
);

-- 이벤트 ↔ 공시 연결 테이블
CREATE TABLE event_disclosures (
    event_id        BIGINT NOT NULL,
    rcept_no        VARCHAR(14) NOT NULL,
    pblntf_ty       CHAR(1),                   -- A~J
    report_nm       VARCHAR(200),
    rcept_dt        DATE,
    role            VARCHAR(20),               -- DECISION, REGISTRATION, COMPLETION, PERIODIC
    PRIMARY KEY (event_id, rcept_no),
    INDEX idx_rcept (rcept_no)
);
```

| `PROGRESS` | 진행 상황 보고 | 자기주식취득상황보고서 |
| `COMPLETION` | 완료/종료 보고 | 합병등종료보고서 |
| `PERIODIC` | 정기보고서 내 사후 기재 | 사업보고서 배당사항, 공모자금사용내역 |
| `CORRECTION` | 정정 공시 | [기재정정]유상증자결정 |

### 7.5 제품별 활용 방식

| 제품 기능 | 그룹핑 필요 여부 | 처리 방식 |
|---|---|---|
| **뉴스 피드** | 필요 | 이벤트 그룹 내 대표 공시 1건만 노출. 후속 공시는 "관련 공시" 링크 |
| **푸시 알림** | 부분 필요 | DECISION 단계만 푸시. PROGRESS/COMPLETION은 별도 알림 또는 무시 |
| **히스토리 추적** | 불필요 | 모든 공시를 시계열로 전부 표시. 이벤트 그룹은 타임라인 UI의 구간 표시용 |
| **투자정보 추출** | 필요 | 이벤트 그룹 내에서 가장 상세한 공시(보통 DECISION)에서 수치 추출 |
